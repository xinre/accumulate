服务间通信的三种方式

1. HTTP（REST API）

是什么？

就是跟你理解的一样，基于 HTTP 协议，通过 JSON 或 XML 格式传递数据。

调用方 (Client)  ──HTTP请求(JSON)──>  被调用方 (Server)
                    <──HTTP响应(JSON)──


特点





前后端通信用的 REST API 本质上就是一种 HTTP 服务



服务间调用也可以用相同的模式，只是数据格式可能更统一



常用的框架：Spring Boot、Express、Flask 等

优点 ✅





简单易懂，生态成熟



调试方便，浏览器、Postman 都能直接测试



支持多种数据格式（JSON、XML）



防火墙友好，HTTP 端口大家都放行

缺点 ❌





每次请求都要建立连接，有一定开销



JSON 序列化/反序列化慢，数据体积大



是同步的，调用方必须等待响应



2. gRPC

是什么？

Google 开发的远程过程调用框架，使用 Protocol Buffers（ProtoBuf） 作为数据格式。

调用方 (Client)  ──HTTP/2 (二进制)──>  被调用方 (Server)
                      <──流式返回──


核心特点





数据用 ProtoBuf 序列化，体积小、速度快



基于 HTTP/2，支持双向流和多路复用



需要先定义 .proto 文件来描述接口

// user.proto
service UserService {
    rpc GetUser (UserRequest) returns (UserResponse);
    rpc GetUserStream (UserRequest) returns (stream UserResponse);  // 流式
}


优点 ✅





性能极高，比 JSON 快 5-10 倍，体积小 3-10 倍



支持流式通信（服务端流、客户端流、双向流）



强类型，有代码生成，IDE 能自动补全



HTTP/2 多路复用，一个连接处理多个请求

缺点 ❌





学习成本高，需要写 proto 文件



二进制格式，调试不如 JSON 直观



浏览器直接调用麻烦（需要 gRPC-Web 或代理）



生态没有 REST 那么广泛



3. 消息队列

是什么？

通过一个"中间件"来传递消息，发送方和接收方不直接通信。

发送服务  ──发消息──>  消息队列（Kafka/RabbitMQ/RocketMQ）  ──收消息──>  接收服务


常见产品







产品



特点



适用场景





Kafka



高吞吐，支持海量消息



日志、大数据、流处理





RabbitMQ



功能丰富，灵活路由



中小型系统，复杂路由





RocketMQ



阿里出品，Java 友好



电商、金融等交易场景

优点 ✅





异步解耦：发送方发完消息就可以干别的，不用等对方处理



削峰填谷：高峰期消息积压，慢慢处理，不会把服务打挂



系统解耦：两个服务可以完全独立，一个挂了不影响另一个



易于扩展：加消费者就能提升处理能力

缺点 ❌





不是实时：消息有延迟，不适合需要即时响应的场景



复杂度高：需要维护消息队列服务



消息顺序/丢失：需要额外处理（事务消息、死信队列等）



调试困难：异步链路追踪比同步调用麻烦

Go 微服务间通信选择

先说结论

简单优先：先用 HTTP

性能瓶颈了 → 换成 gRPC

需要异步 → 加消息队列

不要一开始就搞全套




具体场景选择

1. HTTP 适合的场景

✅ 需要被外部调用（其他语言、其他团队）
✅ 接口简单、调用量不大（几千 QPS 以下）
✅ 调试为主，不想搞复杂
✅ 你的服务是对外的网关/入口


Go HTTP 框架选型：

// Gin - 最流行，简单够用
r := gin.Default()
r.GET("/user/:id", getUser)

// Fiber - 性能更好，对标 Express
app := fiber.New()
app.Get("/user/:id", getUser)


典型用例：





API 网关



外部暴露的 HTTP 接口



内部简单的 CRUD 调用



调用量 < 5000 QPS



2. gRPC 适合的场景

✅ 内部服务调用，追求极致性能
✅ 需要流式传输（直播、监控、聊天）
✅ 服务间调用频繁，QPS 很高
✅ 想要强类型、自动生成代码


Go gRPC 框架：

// 定义 proto
// protoc --go_out=. --go-grpc_out=. user.proto

// 服务端
type UserServer struct {
    pb.UnimplementedUserServiceServer
}

func (s *UserServer) GetUser(ctx context.Context, req *pb.UserRequest) (*pb.UserResponse, error) {
    return &pb.UserResponse{Name: "张三"}, nil
}

// 客户端
conn, _ := grpc.Dial("localhost:50051")
client := pb.NewUserServiceClient(conn)
resp, _ := client.GetUser(ctx, &pb.UserRequest{Id: 1})


典型用例：





服务间高频调用（用户服务频繁查订单）



数据同步（用户信息变更通知）



流式场景（实时推送、监控数据）



QPS > 5000，性能敏感



3. 消息队列适合的场景

✅ 不需要立即知道结果
✅ 需要异步解耦
✅ 有流量峰值需要削峰
✅ 多个服务需要接收同一消息

实际项目建议

简单项目（1-5个服务）

只用水泥砖：

加钢筋：
